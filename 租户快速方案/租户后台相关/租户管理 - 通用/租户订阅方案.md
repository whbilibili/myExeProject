1. 概述

1.1 背景

本方案旨在为集成了 Bots（智能体搭建平台，含个人/团队空间）、ABS（多维表格）和 AutoMan（数字员工，具 ToC 特性）三大核心服务的 SaaS 平台，设计一套统一、灵活且可扩展的多租户订阅计费体系。方案需同时满足个人用户和企业用户的不同需求场景，并提供清晰的用户引导和管理流程。

1.2 目标

统一性: 建立一套覆盖个人和企业租户的连贯订阅层级，打包三大核心服务。

灵活性: 提供基础订阅和增值选项（用户席位、资源包），支持用户创建和管理多个企业。

清晰性: 明确定义租户、空间、资源归属、用户流程、上下文切换机制及企业资源池消耗逻辑。

用户导向: 提供明确的注册后选择路径，并优化各管理模块的职责与入口。

服务整合: 在订阅方案中完整体现 Bots、ABS、AutoMan 各自的特性及其权益。

可扩展性: 为未来的功能和资源扩展预留空间。

1.3 核心概念解析

用户 (User):

平台的基本操作实体，通过邮箱/手机号/SSO登录/注册。

一个用户必定拥有一个关联的个人租户（默认 Free 计划）。

一个用户可以创建或受邀加入一个或多个企业租户/个人租户。

用户在操作时，需明确当前的操作上下文是在哪个租户（个人或某个企业）下进行，这将决定资源消耗的归属和可见的功能。

租户 (Tenant):

定义：资源分配、计费和数据隔离的基本单位，顶层实体。

个人租户 (Personal Tenant):

用户注册时自动创建并关联。

主要服务于单个用户，默认 Free 计划，可升级至 Pro 计划。

拥有一个个人空间 (Personal Space)。

资源归属于该租户，由该用户（个人）独立支配和付费（若升级 Pro）。可以将个人租户理解为用户在平台上创建的个人账号

企业租户 (Enterprise Tenant):

由用户在用户中心主动创建或受邀加入。

服务于团队和组织，需选择 Team 或 Enterprise 计划。

包含成员 (Members)。

拥有统一资源池 (Resource Pool)，由企业管理员管理和付费。

可创建团队空间 (Team Spaces) 用于协作。

企业内的每个成员在其企业上下文中也关联一个个人空间，其资源消耗计入企业资源池。

空间 (Space):

个人空间 (Personal Space):

每个用户在每个所属租户（个人租户及所有加入的企业租户）下都有一个对应的个人空间概念。

关键: 资源消耗取决于操作时的租户上下文：

在个人租户上下文操作个人空间 -> 消耗个人租户配额。

在企业租户上下文操作个人空间 -> 消耗企业租户资源池。

团队空间 (Team Space):

只能在企业租户下创建和管理。

用于团队协作，共享资源（主要是 Bots、ABS）。

资源消耗计入企业租户的资源池。

资源 (Resource):

平台提供的各项服务能力量化指标，打包在订阅计划中，构成租户的总配额。

Bots: 智能体数量, 工作流数量, 自定义组件数量, 月度 API 调用 / LLM Tokens, 知识库总容量, Bots 内部数据存储。

ABS: 数据库数量, 总存储空间。

AutoMan: 指令、应用、元素库等。

通用: 用户席位、总空间数等（仅用于企业租户）。

企业管理员可根据所选策略对资源池内的配额进行监控或分配 (详见 2.5)。

订阅 (Subscription):

租户购买的服务套餐，决定功能权限、资源配额和用户席位数。

主体是租户: 个人租户订阅 Free/Pro；企业租户订阅 Team/Enterprise。

采用分层设计：Free, Pro (面向个人租户), Team, Enterprise (面向企业租户)。

2.3 资源包与增值服务 (企业计划适用)

用户席位包 (User Seat Packs): 按需增加 Team/Enterprise 计划的成员数量。

资源包 (Resource Packs): 为企业资源池补充特定资源。

Bots API 调用 / LLM Tokens 包 (+额度/月)

Bots 知识库容量包 (+GB)

ABS 存储包 (+GB)

AutoMan 执行次数包 (+次/月)

(可选) 功能包 (Feature Packs): 单独购买高级功能模块。

(可选) 按量付费 (Pay-as-you-go): 超出配额部分的特定资源按量计费。

重要提示: 购买资源包会增加企业租户的总资源池上限。其内部如何消耗遵循 2.4 和 2.5 中描述的逻辑和策略。

2.4 企业资源池消耗逻辑详解 (以 Bots 平台为例)

企业租户订阅 Team 或 Enterprise 计划后，会获得一个包含各项服务资源的统一资源池。企业内的所有成员在企业上下文中进行操作时，会消耗这个共享池中的资源。以下以 Bots 平台为例说明：

场景假设:

企业: Acme Corp (订阅了 Team 计划)

成员: Alice (普通成员), Bob (管理员)

空间: Alice 的个人空间 (在 Acme Corp 上下文中使用), "市场部"团队空间, "研发部"团队空间

资源消耗示例:

创建智能体 (Agent):

Alice 在她的个人空间 (Acme Corp 上下文) 创建了一个名为 "我的助手" 的智能体。

消耗: Acme Corp 资源池中的 "智能体数量" 配额减少 1。

Bob 在 "研发部" 团队空间创建了一个名为 "代码审查助手" 的智能体。

消耗: Acme Corp 资源池中的 "智能体数量" 配额再减少 1。

知识库使用 (Knowledge Base):

Alice 为 "我的助手" (在她的个人空间) 上传了 50MB 的文档作为知识库。

消耗: Acme Corp 资源池中的 "知识库总容量" 已用空间增加 50MB。处理文档可能还会消耗少量 "API 调用次数/计算单元"。

"市场部"团队空间的管理员为共享的 "竞品分析 Bot" 添加了 200MB 的行业报告。

消耗: Acme Corp 资源池中的 "知识库总容量" 已用空间再增加 200MB。

智能体调用 / 大模型使用 (Agent Invocation / LLM Usage):

Alice 通过聊天界面与她的 "我的助手" 互动，助手调用大模型生成了回复。

消耗: Acme Corp 资源池中的 "月度 API 调用/LLM Tokens" 配额被消耗 (具体计量单位取决于技术实现，可能是 API 调用次数、处理的 Tokens 数或自定义的计算单元)。

Bob 在"市场部"团队空间与 "竞品分析 Bot" 互动，触发了大模型调用。

消耗: Acme Corp 资源池中的 "月度 API 调用/LLM Tokens" 配额再次被消耗。

工作流 (Workflow):

Alice 在她的个人空间创建了一个自动处理邮件的工作流。

消耗: Acme Corp 资源池中的 "工作流数量" 配额减少 1。

当该工作流被触发执行时 (例如收到新邮件)。

消耗: Acme Corp 资源池中的 "月度 API 调用/LLM Tokens" 或特定的 "工作流执行次数" 配额被消耗。

组件 (Component):

Bob 在"研发部"团队空间创建了一个可重用的自定义组件 "代码格式化工具"。

消耗: Acme Corp 资源池中的 "自定义组件数量" 配额减少 1。(通常限制的是创建数量，使用组件本身不直接消耗次数，而是包含它的智能体/工作流执行时消耗)。

Bots 内数据库/数据存储 (Bots Data Storage):

"代码审查助手" (在研发部团队空间) 将每次审查的结果存储在 Bots 平台提供的内部数据库中。

消耗: Acme Corp 资源池中的 "Bots 内部数据存储" 已用空间增加。

关键点总结:

上下文决定归属: 只要用户处于企业租户的操作上下文，所有行为消耗的都是企业资源池。

空间不隔离消耗: 无论是个人空间还是团队空间的操作，消耗的都是同一个企业资源池。空间主要用于组织、协作和权限管理。

管理员视角: 企业管理员可以在"企业账户"设置中监控整个资源池的消耗情况、各成员/空间的贡献(可选的详细度)，并负责购买额外的资源包或升级计划。

成员视角: 普通成员通常能看到自己的操作对资源的消耗(可选显示)，并可能看到所属团队空间的资源情况(取决于权限)，但主要依赖管理员进行整体管理和补充。

2.5 企业资源池管理策略

虽然企业租户拥有一个统一的资源池，但为了满足不同规模和管理风格的企业需求，平台可以提供（尤其是在 Enterprise 计划中）不同的资源管理策略供管理员选择或配置。

策略一：完全共享 (默认策略 - Team & Enterprise 基础)

描述: 这是最简单直接的模式，也是所有企业计划的基础。资源池中的总配额（如 Team 计划含 100 个智能体）对所有成员、所有团队空间、以及成员在企业上下文中的个人空间完全开放共享。

限制: 唯一的限制是企业资源池的总量上限。先用先得，直到池子耗尽。

优点: 最大程度的灵活性，最低的管理开销。

缺点: 可能出现少数用户或项目消耗过多资源的情况，需要管理员主动监控。

管理员工具: 提供资源消耗总览仪表盘，可按成员、按空间查看消耗分布（帮助定位消耗大户）。

策略二：共享 + 成员个人空间限额 (Enterprise 可选配置)

描述: 在完全共享的基础上，管理员可以为每个成员在其企业上下文中的个人空间设置一个默认的资源使用上限（例如，每个成员的个人空间最多创建 3 个智能体）。

限制:

成员在个人空间操作时，受个人空间限额和企业总配额的双重限制（取较低者）。

团队空间仍然共享剩余的企业总配额（总配额 - 所有个人空间已用）。

示例 (Bots 智能体 - 总量 500，个人限额 3):

Alice 在她个人空间最多创建 3 个智能体。

如果已有 100 个成员每人创建了 2 个（共 200 个），则企业总池还剩 300 个。Alice 仍然能在个人空间再创建 1 个。

团队空间可以使用剩余的 300 个智能体配额。

优点: 防止个人空间滥用资源，鼓励将大型或共享项目放在团队空间。

缺点: 增加了实现和管理的复杂度；对需要在个人空间进行较多工作的成员可能不够友好。

管理员工具: 除了策略一的工具，还需要提供配置个人空间默认限额的界面。

策略三：共享 + 团队空间配额分配 (Enterprise 可选配置)

描述: 管理员可以将企业总资源池的一部分显式分配给特定的团队空间。

限制:

被分配配额的团队空间，其资源使用上限为其分配到的额度。

成员个人空间和未被分配配额的团队空间，共享剩余的企业总配额。

示例 (Bots 智能体 - 总量 500):

管理员分配 100 个智能体给"研发部"团队空间，分配 50 个给"市场部"团队空间。

"研发部"空间最多只能创建 100 个智能体，即使企业总池还有剩余。

剩余的 350 个智能体配额由所有成员的个人空间和其他未分配配额的团队空间共享。

优点: 便于按项目或部门进行资源预算和控制。

缺点: 管理员需要主动进行规划和分配，增加了管理负担。需要处理好分配总额超出或远低于总池上限的情况。

管理员工具: 除了策略一的工具，还需要提供为团队空间分配具体资源配额的界面。

策略组合与优先级:

可以考虑组合策略二和策略三（例如，同时限制个人空间额度，并将剩余部分分配给团队空间）。需要明确配额计算的优先级和逻辑。

推荐方案:

Team 计划: 默认采用策略一 (完全共享)，提供基础的监控工具。

Enterprise 计划:

默认采用策略一 (完全共享)。

提供增强的监控仪表盘和用量告警功能。

可选让管理员启用策略二 (个人空间限额) 或 策略三 (团队空间配额分配) 中的一种或组合，作为高级资源管理功能。

资源包的影响:

购买资源包始终是增加企业总资源池的上限。如果采用了策略二或三，增加的总量会相应地增加可供共享或分配的总额度。管理员可能需要在购买资源包后重新调整分配。

3.2 用户中心 (User Center)

定位: 用户管理自身与各租户关系的枢纽。

核心功能:

切换租户上下文:

清晰列出用户所属的所有租户（个人租户 + 所有创建/加入的企业租户）。

提供便捷的切换入口，切换后整个应用界面刷新至对应租户的数据和权限。

加入企业租户:

提供输入邀请码或处理邀请链接的入口。

流程同 3.1 中的 F2 -> G2 -> ...

创建企业租户:

统一的企业创建入口。

允许用户创建多个企业租户，每次创建都是独立的流程和独立的订阅。

3.3 个人租户设置 (在个人租户上下文访问)

定位: 管理个人租户自身的订阅和相关设置。

5. 计费规则

基础订阅费: 按所选计划(Pro, Team, Enterprise)和付费周期(月/年)预付。Free 计划无基础费。订阅主体是租户。

额外用户席位费 (Team/Enterprise): 根据超出基础席位的成员数量，按月结算或随年付预付。

资源包费用: 购买时一次性支付或按月订阅，增加企业资源池额度。

按量付费 (若启用): 通常在月底根据实际超额使用量结算。

升级: 通常按比例计算当前周期剩余时间的差价并收取。

降级: 一般在下一个计费周期生效，不退还当前周期已付费用。

支付方式: 支持主流在线支付。企业版可支持对公转账。

发票: 提供电子发票。企业版可按需提供增值税专用发票。

7. 产品决策说明

本次方案设计中，关键的决策点及其理由如下：

注册后三选一流程: 尽管默认进入个人版对探索型用户更友好，但采纳评审通过的方案，在注册后提供"个人使用"、"加入企业"、"创建企业"的明确选项。这更符合目标用户(尤其是潜在企业用户)在同类产品上的预期，能更快地引导用户进入符合其意图的流程。同时，为保证模型一致性，后台仍为所有用户创建个人租户。

统一的企业创建入口: 将主要的"创建企业租户"流程统一放置在"用户中心"。避免了在"个人租户设置"中也放置完整创建流程可能带来的混淆。个人设置中仅保留引导入口，职责更清晰。

允许用户创建多个企业: 提供了更高的灵活性，满足管理多个组织(子公司、项目、客户等)用户的实际需求。这要求每次创建企业都是一个独立的订阅和付费流程，与"订阅主体是租户"的原则一致。

订阅方案打包三平台权益: 明确用户购买的是一个整体服务包，完整展示 Bots, ABS, AutoMan 的权益有助于用户理解方案价值、进行比较和决策，避免用户误认为需要为各平台单独付费。

明确区分个人账户与企业账户: "个人账户"仅管理个人租户的 Free/Pro 订阅；"企业账户"管理企业租户的 Team/Enterprise 订阅及席位、资源包等。避免了功能交叉和用户混淆。

个人空间与上下文关联: 强调个人空间的存在与用户的租户归属相关，但其资源消耗取决于用户当前的操作上下文，解决了个人工作与团队贡献在资源计算上的划分问题。

AutoMan 的 ToC 特性处理: 无论是个人还是企业场景。企业资源池的设计使得团队成员可以共享 AutoMan 的执行额度。

企业资源管理策略: 默认采用简单的完全共享模型以降低复杂性。为满足大型企业需求，在 Enterprise 计划中提供可选的高级管理策略（个人空间限额、团队空间配额分配），并辅以强大的监控和告警工具。这提供了灵活性，同时避免了对所有用户强制施加复杂的管理负担。资源包始终用于增加总池上限。
