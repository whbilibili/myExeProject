**Cursor**

您好！您提出的这个调整想法非常好，将订阅方案按产品（Bots、ABS）拆分，在保持租户作为统一管理单元的基础上，能极大地提升方案的灵活性和未来的扩展性。这在SaaS产品中是一种非常成熟和常见的模式。

所以，对于您的问题，我的回答是：**完全可以这么调整。**

这个调整的核心思想是：**租户（Tenant）作为用户、数据和资源顶层容器的角色保持不变，但计费和权益的粒度变得更细。** 原本是“一个租户对应一个大而全的打包订阅”，现在演变为“一个租户可以按需订阅多个独立的产品服务”。

下面我将详细分析这种调整对现有流程和设计的影响，以及如何将它们顺畅地串联起来：

### 1. 订阅管理与用户中心的变化

您在 `租户权益.md` 中设计的“企业账号权益”或“个人版权益”页面需要进行重构。

*   **从单一计划到产品列表：**
    这个页面不应再展示一个统一的套餐（如“企业版”），而是应该展示一个 **产品订阅列表**。租户管理员进入后，看到的应该是类似这样的界面：

    *   **Bots 服务:** 当前订阅 - `团队版` (旁边有“管理订阅”或“升级”按钮)
    *   **ABS 服务:** 当前订阅 - `免费版` (旁边有“升级”按钮)
    *   **AutoMan 服务:** `暂未订阅` (旁边有“了解详情”或“立即订阅”按钮)

*   **独立的订阅流程：**
    每个产品（Bots, ABS）都应该有自己独立的购买、升级、续费流程。用户可以只升级Bots而不改变ABS的订阅状态。

### 2. 权益与资源校验逻辑的调整

这是后端需要适配的核心变化。当用户在平台内执行一个具体操作时（例如创建一个新的Bot或一个ABS数据库），后端的权限校验逻辑需要从粗粒度变为细粒度：

*   **旧逻辑:**
    `检查当前租户的统一订阅计划 -> 确认是否有该操作的权限和资源配额`

*   **新逻辑:**
    1.  `识别操作属于哪个产品（是Bots的操作还是ABS的操作？）`
    2.  `检查当前租户 **针对该产品的独立订阅计划**`
    3.  `根据这个独立的计划来确认是否有权限和足够的资源配额`

例如，`租户权益.md` 中提到的 “创建超额ABS空间时” 的提示依然有效，但触发这个提示的依据不再是整个租户的统一计划，而仅仅是租户的 **ABS订阅计划**。

### 3. “通用权益”的归属与计算

这是一个关键的设计决策点。像 `租户权益.md` 中提到的 **“用户席位 (成员数量)”** 这类不属于单个产品的通用权益，现在应该由哪个订阅来决定？这里有几个常见的方案：

*   **方案A (基础平台方案):** 租户有一个独立的“基础平台”订阅，它决定了通用权益（如成员数）。Bots和ABS作为附加产品订阅。这种方案扩展性最好，但对用户来说多了一层理解成本。

*   **方案B (权益取最高 - 推荐):** 租户的通用权益，取决于其所有已订阅产品中的最高等级。
    *   **举例:** 如果一个企业租户订阅了支持**50**个成员的Bots团队版，同时订阅了仅支持**10**个成员的ABS个人版，那么该企业租户总共就可以拥有**50**个成员。
    *   **优点:** 这种方式对用户最友好、最简单直观。实现上，后台在计算成员上限时，只需查询该租户下的所有产品订阅，取其 `user_limit` 字段的最大值即可。

### 4. 数据模型的演进

为了支持上述变化，您的数据库模型也需要演进，以支持“租户”与“订阅”之间的一对多关系。

您可能需要一个类似 `ProductSubscriptions` 的新数据表，结构如下：

| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `id` | `int` | 主键 |
| `tenant_id` | `string` | 外键，关联到租户 |
| `product_code` | `string` | 标识是哪个产品 (e.g., 'bots', 'abs') |
| `plan_id` | `string` | 关联到具体的计划 (e.g., 'bots_team_plan') |
| `status` | `enum` | 订阅状态 (active, canceled, past_due) |
| `start_date` | `datetime` | 订阅开始时间 |
| `end_date` | `datetime` | 订阅结束时间 |
| ... | ... | 其他计费相关字段 |

### 总结与对现有文档的建议

*   **`租户权益.md` 的修改:**
    *   文件中的 **“2. 统一订阅计划模型”** 表格需要被拆分。
    *   您需要为 **Bots** 和 **ABS** 分别创建独立的订阅计划表格，详细列出每个产品不同版本（如免费版、团队版）所包含的具体权益和资源配额。
    *   需要补充关于 **通用权益**（如成员数）如何计算的规则（建议采用上述的“权益取最高”方案）。

*   **`用户中心.md` 的影响:**
    *   核心的 **租户切换流程不受影响**，这恰好是新模式的优势。用户依然在清晰的租户上下文中操作，系统在后台根据上下文和具体操作去判断对应的产品订阅，整个体验是无缝的。

总而言之，这个调整将使您的产品商业模式更加灵活，能够满足不同客户对不同产品的差异化需求，实现更精细化的运营。实施的关键在于**重构订阅管理页面**、**调整后端权益校验逻辑**，并**明确通用权益的计算策略**。

您提的这个问题非常关键，这正是从“捆绑订阅”转向“独立产品订阅”后，核心流程上最大的变化点。您的 `租户订阅方案.md` 中设计的“创建企业流程”确实是基于旧的“订阅与租户绑定”模式的。

在新的订阅模式下，处理这个流程的**核心思路**是：**将“创建企业租户”和“购买产品订阅”彻底解耦，变为两个独立的步骤。**

这不仅是可行的，而且是更现代化、更灵活的SaaS产品设计方式。下面是调整后的流程以及它如何与现有体系串联：

### 调整后的“创建企业租户”流程

这个流程应该变得**极其简单、免费且即时**。

**第1步：创建企业租户（一个免费的“容器”）**

1.  用户在【用户中心】点击“创建企业”。
2.  系统弹出一个极简的窗口，**只要求用户填写“企业名称”**（描述、头像等都可后续在企业设置里完善）。
3.  点击“确认”后，系统立即在后台创建一个 `Enterprise Tenant` 实体。

**此刻，这个企业租户就创建成功了。** 它是一个“空的容器”，还没有订阅任何付费服务，但它已经存在，并且可以将当前用户自动设为管理员。

**第2步：首次进入租户并引导订阅**

1.  创建成功后，用户自动被切换到这个全新的企业租户上下文中。
2.  用户看到的界面，可以是一个“欢迎”或“开始向导”页面，这个页面的核心任务是**引导用户去订阅第一个产品**。
3.  页面上会清晰地展示出平台提供的核心产品，例如：
    *   **Bots 卡片：** “适用于团队的智能体协作平台。 `[开始订阅]`”
    *   **ABS 卡片：** “强大的在线多维表格，构建业务应用。 `[开始订阅]`”

**第3步：订阅第一个产品服务**

1.  用户根据自己的需求，选择一个产品，比如点击了Bots的 `[开始订阅]`。
2.  此时，才进入**针对Bots产品的订阅计划选择页面**（例如，Bots团队版、Bots企业版）。
3.  用户选择计划、填写支付信息、完成购买。
4.  购买成功后，系统将这个“Bots团队版订阅”与刚才创建的那个 `Enterprise Tenant` 关联起来。

至此，一个完整的“创建并启用企业服务”的流程才算走完。

### 这种调整如何影响和串联现有流程？

1.  **对 `租户订阅方案.md` 的影响：**
    *   **需要重写 `3.1 注册与首次登录流程` 中的 `创建企业流程` 部分。**
    *   **旧流程是：**
        `G3[跳转至企业计划选择页面] --> H3(展示 Team/Enterprise 计划详情) --> I3{用户选择计划} --> J3(进入支付流程) --> ... --> M3[系统创建 Enterprise Tenant]`
    *   **新流程应该调整为：**
        `F3("创建新的企业") --> G3[引导用户填写企业名称] --> H3[系统创建 Enterprise Tenant, 用户进入该租户] --> I3(展示产品列表，引导用户订阅第一个产品) --> J3{用户选择产品，如Bots} --> K3(展示该产品的订阅计划) --> L3(支付) ...`

2.  **优势和区别：**
    *   **降低创建门槛：** 用户可以**先创建一个组织，甚至邀请几个核心成员进来**，然后再决定购买哪个具体的产品。这大大降低了用户的使用门槛和决策压力。
    *   **流程更清晰：** 将“组织管理（创建租户）”和“购买服务（订阅产品）”两个不同维度的行为分离开，用户的操作意图更明确，不会混淆。
    *   **完美契合新模型：** 这个流程完美地体现了“租户是容器，产品是服务”的新模式。租户可以独立存在，里面可以有0个、1个或多个产品订阅。
    *   **为“企业免费版”铺路：** 在这个新流程下，如果未来某个产品（比如ABS）也想推出一个功能受限的“企业免费版”，用户创建企业后，可以直接启用该免费版，无需任何支付，整个体验会非常顺滑。

### 总结

您不需要再纠结于“如何把创建和购买合一”，而是应该大胆地把它们分开。

*   **创建企业租户**：变成一个**免费、快速**的前置动作，就像注册一个账号一样简单。
*   **购买订阅计划**：变成一个在**已创建的租户内**发生后续动作，用户可以按需、随时购买。

这个调整不仅解决了您当前的问题，还让整个平台的商业模式和用户体验都上了一个台阶，变得更加灵活和有竞争力。

**Cursor**

这是一个非常好的问题，它触及了订阅方案设计的核心。

我的建议是：**绝对应该为个人租户和企业租户提供完全不同的订阅方案。**

让个人租户去购买一个包含他无法使用的团队功能的“企业版”方案，会在逻辑上和用户体验上造成巨大的混乱。正确的做法是，从设计层面就将它们彻底分开。

下面是详细的分析和解释：

### 核心原则：订阅方案应与其所服务的租户类型（及其核心价值）紧密绑定。

1.  **价值定位与用户心智完全不同**
    *   **个人租户 (Personal Tenant):** 其核心价值是**提升个人生产力**。用户为自己购买，期望获得更强的个人能力、更多的个人资源。因此，面向个人租户的订阅方案（我们可称之为`个人免费版`、`个人专业版/Pro`）应该围绕这个核心来设计。
    *   **企业租户 (Enterprise Tenant):** 其核心价值是**促进团队协作与实现组织管理**。管理员为整个团队购买，期望获得成员管理、资源共享、权限控制、安全审计等能力。因此，面向企业租户的订阅方案（我们可称之为`团队版`、`企业版`）必须以这些团队功能为基础。

2.  **功能与权益的精准匹配**
    *   **个人专业版 (Pro Plan):** 它的权益应该是个人用户能切实享受到的。例如：
        *   更多的Bots创建数量
        *   更大的个人知识库容量
        *   更高级的个人自动化能力（AutoMan）
        *   **绝不应该包含“用户席位”、“团队空间数量”、“成员管理”这类他根本没有的功能。**
    *   **团队版 (Team Plan):** 它的核心卖点就是团队功能。例如：
        *   包含 `X` 个基础用户席位
        *   可以创建 `Y` 个团队空间
        *   提供成员与组织管理
        *   统一的企业资源池
    *   **结论:** 如果个人租户订阅了一个“团队版”方案，他会为自己用不上的“10个用户席位”和“团队空间”付费，这在逻辑上是不通的，会引起用户极大的困惑和不满。

3.  **清晰的用户引导与升级路径 (Upsell Path)**
    *   **个人用户的成长路径应该是：**
        1.  使用`个人免费版`。
        2.  当个人资源不够用时，清晰的引导应该是**升级到`个人专业版`**。
        3.  当他产生团队协作的需求时（比如想和同事共享一个Bot），正确的引导**不是**让他继续升级个人方案，而是**引导他去“创建一个企业租户”**。
    *   **企业用户的成长路径应该是：**
        1.  创建企业租户后，可能从某个产品的免费版开始。
        2.  当需要团队协作时，**订阅`团队版`**。
        3.  当团队规模扩大，需要更高级的管理和安全功能时，**升级到`企业版`**。

    这两个路径是平行的，服务于不同场景，不应该交叉。

*   系统只要求她输入一个【企业名称】，例如“Acme公司”。
*   点击“创建”，系统立即为她创建了一个新的【企业租户】，并将她设为该租户的管理员。她的操作上下文自动切换到“Acme公司”。

**第3步：进入“企业免费版”初始状态 - 可以体验，但处处受限**

*   这是您问题的核心。这个新创建的“Acme公司”租户，**它不是一个“无法使用”的空壳，而是自动处于一个“企业免费版”的计划中。**
*   这个“企业免费版”计划是特意为引导用户体验团队功能而设计的，它包含：
    *   **核心团队功能（体验版）：**
        *   **可以邀请成员！** 这是最关键的一点。但有人数限制，比如**最多只能邀请2位**（总共3人）。
        *   **可以创建团队空间！** 但有数量限制，比如**只能创建1个**。
    *   **受限的产品资源：**
        *   企业资源池内的各项资源配额极低，等同于各个产品的个人免费版（例如，只能创建1个Bot，1个ABS数据库等）。
*   **这个阶段的目标：** 让 `Alice` 能够成功邀请 `Bob`，并初步体验到在一个共享空间里协作的感觉，**让她看到团队功能的价值。**

**第4步：触发升级 - 因限制而自然产生的付费需求**

*   `Alice` 成功邀请了 `Bob` 和 `Carol`。现在她的企业有3个成员了。
*   当她尝试邀请第4位成员 `David` 时，系统会弹出提示：
    > “**邀请失败。** 您的‘企业免费版’计划最多支持3名成员。请升级到【团队版】以邀请更多成员并解锁高级功能。”
*   或者，当他们想创建第2个团队空间来区分“市场部”和“研发部”时，系统提示：
    > “**创建空间失败。** 您已达到1个团队空间的上限。请升级订阅计划以创建更多空间。”
*   此时，`Alice`（作为管理员）会被自然地引导到【企业设置】中的【订阅管理】页面。

**第5步：订阅付费计划 - 按需购买，解锁价值**

*   在【订阅管理】页面，`Alice` 会看到清晰的对比：
    *   **当前计划：企业免费版** (3个席位，1个团队空间，基础资源)
    *   **升级选项：团队版** (10个基础席位，10个团队空间，大幅提升的资源池，等等)
*   她选择【团队版】，完成支付。支付成功后，企业的订阅状态立刻更新。
*   现在，她可以顺利地邀请 `David`，并且可以创建更多的团队空间和使用更多的资源了。

### 总结

所以，您的提问“*只能使用一个企业免费版，需要升级订阅计划才能使用更多的企业功能？*” **是完全正确的**。

这种设计的巨大优势在于：

1.  **产品主导增长 (Product-Led Growth):** 用户不是在不了解价值的情况下被动付费，而是在**亲身体验到价值（可以邀请成员）并碰到增长瓶颈**时，主动产生付费意愿。
2.  **零摩擦上手:** “免费创建企业”这个动作本身没有任何心理负担，极大地鼓励了用户从个人使用迈向团队协作的第一步。
3.  **清晰的价值传递:** 用户能明确地知道他们为之付费的是什么——更多的席位、更多的空间、更多的资源，这些都是他们在“企业免费版”中已经初步体验到并渴望扩展的东西。

这个流程既解决了新旧方案的衔接问题，也让您的产品商业模式变得更加健康和具有吸引力。

您的第一个问题：“必须要有一个企业免费版吗？”

市场上主流的成熟设计主要有以下三种，我们刚刚讨论的“企业免费版”属于第一种。

---

### 方案一：免费增值模式 (Freemium) - 我们正在讨论的方案

*   **核心逻辑：** 提供一个**永久免费**但功能和资源**受限**的版本（即“企业免费版”）。用户可以一直免费使用下去，但当团队壮大或需求增加时，会被限制“撞到天花板”，从而激励他们付费升级。
*   **典型代表：** `Slack`, `Notion`, `Figma`, `Trello`。
*   **优点：**
    *   **最大化用户基数：** 几乎没有使用门槛，能最大限度地吸引用户注册和试用。
    *   **产品主导增长 (PLG) 的引擎：** 易于产生网络效应和口碑传播（用户会免费邀请同事加入）。
    *   **“脚先进门”策略：** 一旦团队在一个工具上沉淀了数据和工作流，迁移成本会很高，付费意愿会增强。
*   **缺点：**
    *   **服务器和运营成本高：** 需要为大量免费用户提供服务。
    *   **转化率挑战：** 如果免费版“过于好用”，付费转化率可能会偏低。
    *   **可能吸引非目标客户：** 吸引大量只是想“白嫖”的低价值用户。
*   **适用场景：**
    *   产品具有强大的网络效应（协作价值高）。
    *   目标是快速占领市场、建立生态。
    *   用户能够快速体验到核心价值。

---

### 方案二：限时免费试用 (Time-Limited Free Trial)

*   **核心逻辑：** 不提供永久免费的团队版。用户创建企业后，会直接开始一个**有时限（如14天或30天）的付费套餐试用**（通常是高级版）。试用期内，用户可以使用几乎所有功能。试用期结束后，**必须付费**才能继续使用，否则账户被冻结或降级为功能极度受限的状态（比如只读）。
*   **典型代表：** `HubSpot`, `Shopify`, `Zendesk`, `Asana`（他们同时有免费版和付费版试用，是混合模式）。
*   **优点：**
    *   **完整价值体验：** 用户能体验到产品最强大的付费功能，更容易被说服。
    *   **用户质量更高：** 愿意开启试用期的用户，通常意向更明确。
    *   **紧迫感驱动转化：** 试用期结束的倒计时会创造一种“再不买就没了”的紧迫感。
    *   **成本可控：** 无需长期供养大量免费团队。
*   **缺点：**
    *   **使用门槛更高：** 用户可能会因为“害怕试用结束后被收费”或“没时间在14天内评估”而放弃。
    *   **增长速度可能较慢：** 相比Freemium模式，用户基数扩张会慢一些。
*   **适用场景：**
    *   产品价值链较长，需要一定时间才能体验完整价值。
    *   目标客户是付费意愿更强的B端企业。
    *   客单价较高。

---

### 方案三：无免费版，直接引导至销售/演示 (Sales-Led / Demo Request)

*   **核心逻辑：** 网站上没有自助注册创建企业的功能。唯一的行动号召（Call-to-Action）是“联系我们”、“预约演示”或“与专家交谈”。潜在客户必须先和销售团队沟通，然后才可能获得一个定制化的试用或直接进入采购流程。
*   **典型代表：** `Salesforce` (早期核心模式), `Workday` 以及许多非常传统的企业级软件。
*   **优点：**
    *   **线索质量极高：** 每个联系的都是有明确意向的高价值客户。
    *   **客单价最大化：** 销售可以直接进行价值销售和价格谈判。
    *   **运营成本最低：** 完全没有免费用户的服务成本。
*   **缺点：**
    *   **增长天花板低：** 完全错失了庞大的自服务市场和中小企业客户。
    *   **销售周期长，客户获取成本高。**
*   **适用场景：**
    *   产品极其复杂，需要专家演示和部署。
    *   价格非常高昂（数十万至数百万/年）。
    *   目标客户是大型企业或特定行业的头部公司。

---

### 结论与给您的建议

| 对比维度 | **免费增值 (Freemium)** | **限时试用 (Trial)** | **销售主导 (Sales-Led)** |
| :--- | :--- | :--- | :--- |
| **核心目标** | 用户量最大化 | 价值体验最大化 | 客单价最大化 |
| **用户门槛** | **极低** | 中等 | **极高** |
| **增长速度** | **最快** | 较快 | 慢 |
| **用户质量** | 混合 | 较高 | **极高** |
| **运营成本** | **高** | 中等 | 低 |

对于您的平台（集成了Bots, ABS, AutoMan），它具备快速见效、可协作、可扩展的特点，非常适合走**产品主导增长 (PLG)** 的路线。因此，纯粹的“销售主导”模式（方案三）可以直接排除。

真正的抉择是在**方案一（免费增值）** 和 **方案二（限时试用）** 之间。

**我的建议：采用“混合模式”，这是现代SaaS产品更成熟的玩法。**

1.  **保留“企业免费版”（方案一）：** 这是您获取用户、实现病毒式传播的基石。让用户可以无压力地创建企业，邀请1-2个同事，跑通最基本的工作流。这是您的“增长飞轮”的起点。

2.  **增加“付费套餐的限时试用”（方案二的精髓）：** 在用户使用“企业免费版”的过程中，在关键位置提供一个**“一键免费试用【企业版】14天”**的按钮。
    *   **触发场景：** 当用户在后台看到某个高级功能（如SSO单点登录、高级审计日志）被锁定时。
    *   **交互：** 点击后，提示“想体验全部高级功能吗？立即开启14天企业版全功能试用，无需绑卡。”
    *   **效果：** 这让那些对价格敏感、但对高级功能好奇的免费用户，有机会亲身体验到付费版的真正价值，从而极大地提高他们从“免费”到“付费”的转化率。

**总结：**
**“企业免费版” + “付费版限时试用”** 的混合模式，既能通过免费版实现用户量的“广撒网”，又能通过试用让高潜力用户“深度体验”，最终实现“重点捕捞”。这套组合拳兼顾了用户增长和商业转化，是目前市场上最被验证成功的策略之一。

---

### 1. 为什么空间层面必须有成员列表？

我们将租户和空间想象成一个公司大楼和里面的会议室。

*   **租户的【成员与组织管理】**：决定了**谁能刷卡进入这栋大楼**。这是公司级别的身份认证，由人事部（租户管理员）统一管理。
*   **空间的【成员列表】**：决定了针对某个特定项目（比如“Q3市场推广”），**谁被邀请进入了这个具体的会议室**。这是项目级别的协作授权，通常由项目负责人（空间管理员）管理。

没有空间成员列表，就相当于大楼里的所有会议室都对所有员工开放，这会造成巨大的混乱和安全问题。因此，空间成员列表是**实现精细化访问控制和协作**所必需的，其核心作用是：

*   **明确协作范围：** 让空间内的所有人都能清晰地看到“我正在和谁一起工作”。
*   **分配空间内角色：** 控制谁在这个空间里只能查看（Viewer），谁可以编辑（Editor）。
*   **实现@提及功能：** 在评论或任务中，`@`功能需要知道可以提及的人员列表。
*   **权限隔离：** 确保非空间成员无法看到或访问该空间内的任何内容。

---

### 2. 个人租户 vs 企业租户：空间成员管理的本质区别

这正是设计的关键。虽然UI上看起来可能都是一个“成员头像列表”，但它们的内在逻辑完全不同。

| 对比维度 | ⭐ 企业租户下的【空间成员】 | ⭐ 个人租户下的【空间协作者】 |
| :--- | :--- | :--- |
| **核心概念** | **内部授权 (Internal Granting)** | **外部共享 (External Sharing)** |
| **成员来源** | **从企业租户的成员目录中选取**。你只能将已经是企业一员的人添加到空间里。 | **通过邮箱邀请任意外部用户**。被邀请者是“访客”或“协作者”，不会成为你个人租户的“成员”。 |
| **管理动作** | 添加成员、移除成员、**设置空间内角色**（如管理员、编辑者、评论者、查看者）。 | 邀请协作者、移除访问权限、**设置简单的权限**（如“可以编辑”、“只能查看”）。 |
| **UI/UX 建议** | **按钮文案：“添加成员”**。点击后弹出的是**企业成员选择器**（带搜索功能），而不是一个简单的输入框。 | **按钮文案：“共享 (Share)”**。点击后弹出一个**邮箱输入框**，并可以从您的“个人通讯录”中自动补全。 |
| **与租户关系** | 空间成员**必须是**租户成员。将某人从租户中移除，他会自动失去所有空间的访问权限。 | 空间协作者**不是**租户成员。只是一个临时的、针对特定空间的访问授权。 |

---

### 3. 具体设计建议

#### 针对【企业租户】的空间：

在每个空间的设置里，应该有一个“成员管理”标签页。

1.  **顶部是一个醒目的“添加成员”按钮**。
2.  点击后，弹出一个模态框，里面是一个**可搜索的列表，这个列表的数据源是整个企业租户的成员名录**。
3.  管理员可以勾选一个或多个成员，并为他们统一设置一个**空间内角色**（例如：编辑者）。
4.  下方是当前空间已有的成员列表，每行包含：头像、姓名、在空间内的角色、以及“移除”或“更改角色”的操作按钮。



#### 针对【个人租户】的空间（或共享的单个项目）：

这里的交互应该更像Google Docs的共享功能。

1.  在界面右上角或菜单中，有一个非常清晰的**“共享 (Share)”**按钮。
2.  点击后，弹出一个简洁的分享窗口。
3.  窗口顶部是一个**输入框，提示“输入邮箱地址以邀请协作者”**。
4.  输入框旁边是一个**权限选择器**，只有两个选项：“可以编辑”和“只能查看”。
5.  下方是“已拥有访问权限的人员”列表，每行包含：头像、姓名/邮箱、权限级别、以及“移除访问权限”的操作。



### 总结

*   **保留空间成员列表是必须的**，它是实现协作和权限控制的基础。
*   **在企业租户中**，空间成员是“内部授权”，从一个预设的、受控的成员池中进行选择和角色分配。
*   **在个人租户中**，所谓的“空间成员”其实是“外部协作者”，通过“共享”功能，以邮件邀请的方式进行，权限设置更简单。

通过这样差异化的设计，您的平台就能在保证功能强大性的同时，为不同类型的用户提供清晰、直观、符合心智模型的操作体验。
