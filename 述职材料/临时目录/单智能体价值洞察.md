1. 产品愿景与核心价值主张

1.1 核心愿景

将智能体详情页从单一的“技术运维监控仪表盘”全面升级为**“业务价值评估与诊断中心”。我们通过构建一套涵盖规模、深度、成本、质量**的四维评估体系，帮助企业管理者、运营专家及开发者实现精益化运营。

1.2 用户故事矩阵 (User Stories)

运营专家: 需通过**转化漏斗（PV → DAU → Session）分析引流效率，结合留存粘性（人均会话）**研判推广价值。

业务负责人: 需通过活跃用户趋势 (DAU) 与 会话深度，评估智能体对实际业务的渗透率与解决问题的能力。

技术/财务: 需通过Token 结构分析与异常用户画像精准定位成本黑洞，实现精细化控费。

开发者: 需通过会话级日志还原和用户反馈，快速复现并修复 Bad Case。

2. 页面布局与信息架构

页面采用 "头部信息栏 + 内容 Tab 分组" 的分层架构，遵循“先总后分、先结论后细节”的认知规律。

2.1 顶部信息枢纽 (Sticky Header)

定位: 静态身份识别区。布局设计: 左侧为大尺寸头像，右侧采用三行式元数据布局，信息密度高但不拥挤。

Row 1 (核心识别):

头像: 48x48px 圆角矩形，支持点击上传/更换。

智能体名称: H1 标题字号，加粗。

状态标签: 胶囊样式。🟢 已发布 (Online) / 🟡 调试中 (Draft) / 🔴 已下线 (Offline)。

Row 2 (归属与元数据):

ID: 智能体 ID (显示前 8 位，Hover 显示全量，支持一键复制)。用于技术排查。

类型: 智能体类型 (Tag: 提示词工程 / 工作流编排 / 自主任务)。

创建者: Owner (头像+姓名，点击跳转至该用户的个人主页)。

归属租户: 所属租户 (成本归属对象，用于财务核算)。

Row 3 (时间戳审计):

创建于: Created Date (YYYY-MM-DD)。

更新于: Last Updated (配置最后修改时间，用于判断数据波动是否由配置变更引起)。

最后运行: Last Used Date (精确到分钟，判断智能体是否“活跃”或“僵尸”)。



2.2 内容分组 (Tab Navigation)

采用标准 Tab 组件，支持点击切换，URL 参数联动（刷新页面后停留在当前 Tab）。

数据概览 (Overview): 全局 KPI 矩阵、五大分区趋势分析、高价值用户透视。（默认选中）

会话日志 (Session Logs): 业务视角的完整对话流水，支持上下文还原与反馈追踪。

用户审计 (User Audit): 以“人”为维度的聚合统计表（活跃/成本/留存分析）。

调用日志 (Trace Logs): 技术视角的 LLM/Tool 调用流水，用于 Token 消耗审计与错误排查。

配置快照 (Config Snapshot): 当前生效版本的配置只读视图，用于对比数据与配置的关系。

3.4 成本与性能 (Cost & Performance)

面向对象: 技术负责人、财务 设计目标: 成本归因与性能监控。

A. 核心指标卡片 (关键指标对)

此区域采用“主指标+副指标”形式展示，强化因果关联。

卡片：单轮对话经济性

主: 平均单轮对话成本 (Avg Cost/Call)

副: 平均每轮 Token (Avg Tokens/Call)

价值: 监控微观成本。若金额涨Token没涨，是模型贵了；若都涨了，是Prompt长了。

卡片：会话经济性 (ROI)

主: 平均会话成本 (Avg Cost/Session)

副: 平均会话 Token (Avg Tokens/Session)

价值: 解决一个用户问题的平均成本。

卡片：Token 结构

主: 输入/输出 Token 比 (I/O Ratio)

副: Input: 222M | Output: 111M 

价值: 识别“阅读型”或“创作型”任务特征。

卡片：技术健康度

主: 成功率 (Success Rate)：成功对话轮次/总对话轮次。

失败定义： 针对每轮对话，所有非业务成功（非HTTP 200或有明确error flag）的调用。

副: 平均响应时间 (Latency)：

定义： 服务端完整处理一次对话请求的平均耗时（从收到请求到完成所有 Token 输出）。

单位： 毫秒 (ms)。

统计范围： 仅统计状态码为 200 的成功请求。

影响因素： 主要受 模型推理速度 (Model Inference) 和 回复内容长度 (Output Tokens) 影响。如果发现 Latency 激增，请优先检查是否是 Output Token 变长了，或者是使用了更复杂的模型（如 GPT-4）。

价值: 稳定性底线监控。

卡片：主要消耗源 (Cost Drivers) 🆕

主 (模型): 花费最多模型 (Primary Model Used) (如: GPT-4)

副 (流): 花费最多工作流 (Linked Workflows) (如: SearchFlow)

价值: [归因] 快速识别成本主要来源。直接回答“为什么这么贵”。

卡片：平均首字延迟

主数值：xxx ms

定义：从发送请求到看到第一个字符的时间。如果该指标过高，请检查“知识库检索耗时”或“工具执行耗时”。

价值: 平均首字延迟越短，用户的主观等待焦虑越低。

B. 可视化图表

图表 6: 成本结构演进 (Cost & Token Structure)

图表类型: 堆叠面积图 (Input/Output Token) + 折线图 (总花费)。

X 轴 (维度): 时间 (日期)。

Y 轴 (左 - 数量): Token 消耗量。系列：Input Tokens (堆叠) + Output Tokens (堆叠)。

Y 轴 (右 - 金额): 总花费 (Total Cost)。展示形式：折线图。

分析意图: 归因分析，确定成本上涨是源于 Token 用量（量涨）还是模型单价（价涨）。

图表7：🚨 性能与错误率关联分析 (Performance Correlation)

类型: 组合图 (双轴 + 多线)。

X轴: 时间 (Time/Date)。

Y轴 (左): 成功率 (Success Rate) [柱状图]，单位 %。

图表类型: 柱状图 (Bar)。

颜色映射 (关键):

若 Success Rate >= 95%: 显示为 浅蓝色/绿色 (健康)。

若 Success Rate < 95%: 该时间点的柱子自动变为 红色 🔴 (报警)。

Y轴 (右): 延迟指标 (Latency Metrics)

图表类型: 折线图 (Line)。

线条 1 (实线):平均响应时间 (Avg Latency) [折线图]，单位 ms。

线条 2 (虚线): 平均首字延迟 (Avg TTFT)。

目的: 快速发现因延迟过高导致的系统报错，或识别系统抖动时段。

首字延迟

Time To First Token

技术定义：从客户端（用户）发出请求开始，到收到服务端返回的**第一个数据包（First Chunk/Token）**并在屏幕上渲染出第一个字符所消耗的时间。

用户感知：用户点击“发送”后，到看到屏幕上“蹦出第一个字”之前的空白等待时间。

在总耗时相同的情况下，TTFT 越短，用户的主观等待焦虑越低。

显示时间，如：1200ms

需增加Tooltip 解释：从发送请求到看到第一个字符的时间。如果该指标过高，请检查“知识库检索耗时”或“工具执行耗时”。

耗时

latency

服务端处理时长。

从系统向模型服务商（如 OpenAI/Azure）发起 HTTP 请求的那一刻开始，到系统接收到模型返回的最后一个字节（Last Token）并断开连接的那一刻结束的时长。

关键指标。如 1200ms。

预警：若 > 5000ms，文字标橙或加⚠️图标。

定义：仅统计模型推理阶段的耗时（Time to Last Token）。

排除项：不包含 平台自身的工作流调度、前置处理、插件联网等待及后置数据清洗的时间。

目的：精准反映模型供应商的响应性能，用于评估“模型是否卡顿”及“生成内容是否过长”。
